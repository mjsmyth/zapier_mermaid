# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Purpose

This repo processes Zapier JSON export files and generates Mermaid diagram documentation for Zaps. The main script parses the JSON, extracts node/step metadata, and renders it as flowchart Markdown via a Mustache template.

## Commands

```bash
# Generate diagrams from a Zapier export
python3 create_zapier_mermaid.py zapfile_20260526_01

# Scan a JSON export for (api, action) pairs missing from the YAML config
python3 scan_zap_config.py zapfile_20260526_01/zapfile_20260526_01.json           # dry-run: report only
python3 scan_zap_config.py zapfile_20260526_01/zapfile_20260526_01.json --write   # write stubs to YAML (auto-backs up first)
```

Dependencies: `pystache` (Mustache templating) and `pyyaml`. Install with:
```bash
conda install pystache pyyaml
```

There are no automated tests or linting tools configured.

## File structure

Input exports live in dated folders at the repo root. Pass the folder path to the script; it looks for the JSON file in this order:

1. `zapfile.json` (generic name, checked first)
2. `<folder-name>.json` (must match the folder name exactly)

```
zapfile_20260526_01/
    zapfile.json              # preferred, or
    zapfile_20260526_01.json  # fallback
```

Output is written to `output/<zap_folder_name>/zapier-<title-slug>.md` â€” one file per active Zap. Zaps with `"status": "off"` are skipped; the slug is the lowercased, hyphenated Zap title. Warnings and errors are logged to `zapier_mermaid.log`.

`zapier_mermaid.md` is a hand-crafted reference example of ideal output (not generated by the script). `output/output.txt` is a manual debug dump of the `zap_doc` dict and is not produced by any script.

## Zapier JSON structure

All JSON files share the same schema:
```
{
  "metadata": { "version": "gdpr_v1" },
  "zaps": [
    {
      "id": <int>,
      "title": <str>,
      "status": "on" | "off",
      "nodes": {
        "<node_id>": {
          "parent_id": <int|null>,   // null = trigger (root node)
          "root_id": <int|null>,     // null = trigger; otherwise points to trigger node
          "action": <str>,           // e.g. "file_in_folder_v2", "util_line_item"
          "selected_api": <str>,     // e.g. "GoogleDriveCLIAPI@1.18.2"
          "type_of": "read"|"write",
          "meta": {
            "stepTitle": <str>,
            "parammap": { ... }      // human-readable param labels
          },
          "params": {
            "my_dict": { ... }       // key-value pairs; values may reference prior steps
          }
        }
      }
    }
  ]
}
```

**Step references:** `params.my_dict` values use `{{=gives["<node_id>"]["<field>"]}}` syntax to reference outputs from earlier steps. These field names are extracted as `step_inputs` and become edge labels in the diagram. The extraction regex matches **exactly 9-digit node IDs** â€” Zapier node IDs outside that range will be silently missed.

## Architecture

### Data flow

```
JSON file â†’ zap_dict["zaps"]
  â””â”€ skip zaps with status "off"
  â””â”€ process_zap(zap, zap_doc, output_path)
       â””â”€ process_node() per node â†’ populates zap_doc["nodes"][node_id]
            â”śâ”€ extracts: parent_id, root_id, stepTitle, selected_api, action, parammap
            â”śâ”€ dispatches to NODE_CONFIG / _SPECIAL_HANDLERS for app-specific fields
            â””â”€ step_inputs: field names from gives[...] references â†’ become edge labels
       â””â”€ build_render_context(zap_doc) â†’ flat context dict for Mustache
            â”śâ”€ assigns sequential IDs: N1, N2, N3, ...
            â”śâ”€ collapses BranchingAPI/filter nodes onto edges (not rendered as nodes)
            â””â”€ sorts branch path edges: "success/yes" first, "error/no" last
       â””â”€ pystache.render(TEMPLATE, context) â†’ writes output/zapier-<slug>.md
```

The `parent_id` chain defines graph edges. `step_inputs` on a child becomes the label on the edge from its parent.

**`parammap` round-trip:** `process_node` stores `parammap` as a `str(dict)` Python representation; `_build_node_label` later recovers it with `ast.literal_eval`. If a `parammap` value contains non-Python-literal content, the eval falls through silently and the parammap line is omitted from the label.

### Config-driven node dispatch (`zapier_node_config.yaml`)

All known `(api_name, action)` pairs are registered in `zapier_node_config.yaml`. The script loads this into three module-level dicts:

- `NODE_CONFIG` â€” keyed on `(api_name, action)`; each entry has `display_name`, `node_type`, `params` (field extractions), `constants`, and optionally `special`
- `NODE_CONFIG_META` â€” keyed on `api_name`; holds `provider` and `emoji` for label building
- `PROVIDER_COLORS` â€” keyed on provider name; holds the Mermaid `style` string

`_normalize_api()` strips the `@version` suffix from `selected_api` before lookup (e.g. `GoogleDriveCLIAPI@1.18.2` â†’ `GoogleDriveCLIAPI`). Unrecognised `(api, action)` pairs log a warning but don't abort.

**To add a new Zapier integration:** add an entry to `zapier_node_config.yaml` under the api name. No Python changes are needed unless the handler logic must be conditional (e.g., checking action type or iterating a sub-dict). In that case, add a function to `_SPECIAL_HANDLERS` in `create_zapier_mermaid.py` and reference it via `special: _your_handler_name` in the YAML. `_SPECIAL_HANDLERS` is a dict mapping handler name strings to functions, populated at module level after the handler functions are defined.

**YAML style values:** Each provider's `style` string must start with a valid CSS property like `fill:` â€” a leading word before `fill:` will produce invalid Mermaid CSS.

### `scan_zap_config.py` â€” config gap detector

Use this when adding a new JSON export that contains integrations not yet in the YAML. It compares all `(api, action)` pairs in the file against the config and reports gaps.

- **Dry run** (default): prints a report and suggested YAML stubs to stdout.
- **`--write`**: mutates `zapier_node_config.yaml` in-place; saves a timestamped `.bak` backup first (e.g. `zapier_node_config.yaml.bak.20260526_143012`).
- **Scans all zaps including `"off"` ones** â€” unlike the main script, which skips disabled zaps.

Stubs are written with `display_name` inferred from camelCase splitting, `node_type` inferred from `type_of`/`parent_id`, and `params: []`. These are starting points â€” review and fill in `params` extractions and correct the `emoji` before running the main script.

### Branch node collapsing

`BranchingAPI` filter nodes are not rendered as graph nodes â€” they are collapsed onto edges by `build_render_context()`. When a node's parent is a branch filter, the edge is drawn from the filter's parent instead, with the branch path label (e.g. "success", "error") as the edge label. Branch path edges are sorted: "success/yes" variants first, "error/no" variants last; ties are broken by `path_eval_index` from the raw params.

### Node label construction (`_build_node_label`)

Labels are assembled as `<br/>`-joined parts:
1. `{emoji} {stepTitle}` (or `display_name` if stepTitle is blank)
2. `[Provider]` â€” appended only on the **first** node of each provider within a diagram (tracked via a `seen_providers` set passed through the loop)
3. `(FirstParamKey: FirstParamValue)` â€” from `parammap`, first entry only
4. The pre-built `params` string (e.g. `"Params: Doc name, Category, URL"`) if present

### Output format (`zapier_mermaid.mustache`)

Each output file is Markdown with YAML frontmatter (title, deprecated, hidden, robots), a summary table, and a fenced `mermaid graph TD` block. Regular nodes use `[" "]` syntax; branch/path nodes use `{" "}` (diamond shape). Edges use `-->|label|` with `step_inputs` as the label. Each node gets a `style` directive with provider-specific fill color from `PROVIDER_COLORS`.

The template uses `{{{triple braces}}}` for unescaped output (required for Mermaid syntax and `<br/>` tags) and `{{double braces}}` for HTML-escaped content. Use triple braces for any field that may contain Mermaid characters or HTML.
